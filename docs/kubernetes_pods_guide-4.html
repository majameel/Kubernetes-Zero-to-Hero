‚öñÔ∏è Docker vs Kubernetes: Container Deployment

üê≥ Docker Approach

Command-Line Arguments:

# Docker run with arguments
docker run -d \
  --name nginx \
  -p 80:80 \
  -v /data:/var/data \
  --network bridge \
  nginx:1.14.2

Characteristics:

  • Direct container deployment
  • Command-line configuration
  • Immediate execution
  • Manual configuration

‚ò∏Ô∏è Kubernetes Approach

YAML Specification:

apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.14.2
    ports:
    - containerPort: 80
    volumeMounts:
    - name: data-volume
      mountPath: /var/data

Characteristics:

  • Pod wrapper deployment
  • YAML configuration
  • Declarative approach
  • Standardized specification

üí° Why YAML Files?

Kubernetes is an enterprise-level platform that provides:

  • Declarative capabilities: Describe desired state
  • Standardization: Consistent format across all resources
  • Version control: YAML files can be stored in Git
  • Documentation: Self-documenting infrastructure

‚ú® Advantages of Multiple Containers in Same Pod

üåê Shared Networking

  • Containers communicate via localhost
  • Container A ‚Üí Container B: localhost:3000
  • No need for complex networking setup
  • All containers share the same Pod IP

üíæ Shared Storage

  • Containers can share files directly
  • Common volume mounts
  • Data synchronization between containers
  • Simplified data exchange

‚ö†Ô∏è Important Note

Multiple containers in a Pod is a rare and advanced use case. We'll cover sidecar and init containers in advanced topics like service mesh. For now, focus on single-container Pods!

üåê Pod Networking

How Pod IP Addressing Works:

  • Kubernetes assigns cluster IP to the Pod (not individual containers)
  • kube-proxy generates this cluster IP (from previous architecture lesson)
  • Applications accessed via Pod IP + container port
  • Pod acts as network boundary for all containers inside

üöÄ Deploy Your First Pod

üìÑ Creating Pod YAML

üí° Finding Pod Examples

Search: "pod kubernetes documentation" to find official examples. You don't need to memorize YAML structure - everyone uses examples and modifies them!

üìã Complete Pod YAML Example

apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.14.2
    ports:
    - containerPort: 80

üîç YAML Structure Explained

Understanding Each Field:

apiVersion: v1 Kubernetes API version for Pod
kind: Pod Type of Kubernetes resource
metadata.name Name of your Pod (choose anything)
spec.containers List of containers in this Pod
containers.name Name of container inside Pod
containers.image Docker image to use
containerPort Port your app listens on (CRITICAL!)

‚ö†Ô∏è Critical Configuration

containerPort MUST match your application's listening port!

  • Nginx: port 80
  • Node.js app: port 3000, 8000, etc.
  • Python Flask: port 5000
  • Java Spring Boot: port 8080

üîÑ Docker Command vs Pod YAML Comparison

üê≥ Docker Equivalent

# Docker command that does the same thing
docker run -d \
  --name nginx \
  -p 80:80 \
  nginx:1.14.2

Command-line arguments translated to YAML fields

‚ò∏Ô∏è Kubernetes Pod YAML

apiVersion: v1
kind: Pod
metadata:
  name: nginx        # --name nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.14.2  # nginx:1.14.2
    ports:
    - containerPort: 80  # -p 80:80

Structured specification in YAML format

üìù Enhanced Pod Specifications

apiVersion: v1
kind: Pod
metadata:
  name: enhanced-nginx
  labels:
    app: nginx
    environment: development
spec:
  containers:
  - name: nginx
    image: nginx:1.14.2
    ports:
    - containerPort: 80
    env:
    - name: ENVIRONMENT
      value: "development"
    resources:
      requests:
        memory: "128Mi"
        cpu: "100m"
      limits:
        memory: "256Mi"
        cpu: "200m"
    volumeMounts:
    - name: nginx-storage
      mountPath: /usr/share/nginx/html
  volumes:
  - name: nginx-storage
    emptyDir: {}

üîß Advanced Pod Features

The enhanced YAML shows additional Pod capabilities:

  • Environment variables: Pass config to containers
  • Resource limits: Control CPU and memory usage
  • Volume mounts: Persistent or shared storage
  • Labels: Organize and select pods
üîç Debugging Pods

üõ†Ô∏è Essential Debugging Commands

When things go wrong with your Pods, these two commands are your best friends:

üìã kubectl logs

# View Pod logs
kubectl logs nginx

# Follow logs in real-time
kubectl logs -f nginx

# Logs from specific container (multi-container Pod)
kubectl logs nginx -c container-name

# Previous container logs (if crashed)
kubectl logs nginx --previous

Purpose: Shows application output, errors, and debugging information from inside your container.

üîç kubectl describe

# Detailed Pod information
kubectl describe pod nginx

# Shows:
# - Pod configuration
# - Events and status changes
# - Resource allocation
# - Error messages
# - Conditions and states

Purpose: Provides comprehensive Pod status, events, and troubleshooting information.

üíº Interview Answer:
"To debug Pods, I use kubectl logs to check application output and kubectl describe pod to analyze Pod status and events."

üöÄ From Pods to Production

üéØ Why Learn Pods First?

While you rarely deploy Pods directly in production, understanding Pods is crucial because:

  • Deployments are Pod wrappers - they contain Pod templates
  • StatefulSets use Pod specifications - for stateful applications
  • DaemonSets deploy Pods - one per node
  • Jobs and CronJobs run Pods - for batch workloads

üìà Production Kubernetes Resources

Pods (Foundation)
Deployments (Day 34)
Services
Auto Scaling & Healing

üéÅ Direct Pod Deployment

apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.14.2

Limitations:

  • No auto-healing if Pod dies
  • No auto-scaling capabilities
  • Manual management required

üöÄ Deployment (Tomorrow's Topic)

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.14.2

Advantages:

  • Auto-healing: Restarts failed Pods
  • Auto-scaling: Adjusts replica count
  • Rolling updates: Zero downtime deployments

üîç Notice the Similarity

Deployment YAML contains a template section that looks exactly like Pod YAML! This is why understanding Pods is essential - they're the building blocks for all other Kubernetes resources.
