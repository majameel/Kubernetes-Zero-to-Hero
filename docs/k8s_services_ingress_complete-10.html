üèóÔ∏è Ingress Concepts

üìÑ Ingress Resource

  • Defines routing rules
  • Created via YAML manifest
  • Specifies hosts, paths, backends
  • Does nothing without Controller
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: example-ingress
spec:
  rules:
  - host: food.bar.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: my-service
            port:
              number: 80
VS

‚öôÔ∏è Ingress Controller

  • Implements the logic
  • Runs as pods in cluster
  • Watches Ingress resources
  • Configures load balancer

Popular Controllers

  • Nginx Ingress
  • HAProxy Ingress
  • F5 BIG-IP
  • Ambassador
  • Traefik

üõ£Ô∏è Ingress Routing Examples

üåê Host-Based Routing

Route traffic based on hostname in request:

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: host-based-ingress
spec:
  rules:
  - host: food.bar.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: http-svc
            port:
              number: 80

üß™ Demo Commands:

# This fails - no Host header
curl <Ingress_IP>
# Result: 404 - Host not found

# This works - correct Host header
curl -H "Host: food.bar.com" <Ingress_IP>
# Result: Success! Routes to http-svc

üìÅ Path-Based Routing

Route traffic based on URL path:

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: path-based-ingress
spec:
  rules:
  - host: food.bar.com
    http:
      paths:
      - path: /first
        pathType: Prefix
        backend:
          service:
            name: http-svc
            port:
              number: 80
      - path: /second
        pathType: Prefix
        backend:
          service:
            name: miyab-svc
            port:
              number: 80

üß™ Demo Commands:

# No path - fails
curl -H "Host: food.bar.com" <Ingress_IP>
# Result: 404

# Correct path - works
curl -H "Host: food.bar.com" <Ingress_IP>/second
# Result: Routes to miyab-svc

curl -H "Host: food.bar.com" <Ingress_IP>/first
# Result: Routes to http-svc

üåü Wildcard Hosts

        backend:

            name: http-svc
            port:
              number: 80

üß™ Demo Commands:

              number: 80

Create auth secret with htpasswd:

üîí TLS Security with Ingress

üö™ SSL Passthrough

Client (HTTPS)
‚Üì
Load Balancer
(Pass-through)
‚Üì
Service (HTTPS)

‚úÖ Advantages:

  • End-to-end encryption
  • No decryption at LB

‚ùå Disadvantages:

  • No L7 features (path routing, WAF)
  • Service handles decryption load
  • Higher latency during high traffic
  • Security risk - no packet inspection
üéØ Use Case: Only for L4 load balancing needs

üì§ SSL Offloading

Client (HTTPS)
‚Üì
Load Balancer
(Decrypt)
‚Üì
Service (HTTP)

‚úÖ Advantages:

  • Reduces service CPU load
  • Lower application latency
  • Full L7 features available
  • WAF and routing work

‚ùå Disadvantages:

  • MAJOR SECURITY RISK
  • Unencrypted LB ‚Üí Service traffic
  • Man-in-the-middle attack risk
  • Internal network exposure
‚ö†Ô∏è NOT RECOMMENDED for security-critical applications

üåâ SSL Bridging (Re-encrypt)

Client (HTTPS)
‚Üì
Load Balancer
(Decrypt + Re-encrypt)
‚Üì
Service (HTTPS)

‚úÖ Advantages:

  • End-to-end encryption
  • Full L7 features available
  • Packet inspection possible
  • WAF protection
  • Security + functionality

‚ùå Disadvantages:

  • Service still handles decryption
  • Some latency impact
  • More complex setup
üéØ RECOMMENDED for production applications

üîë TLS Certificate Handling

Store Certificates in Kubernetes Secrets

# Create TLS secret
kubectl create secret tls food-bar-tls-secret \
  --cert=path/to/cert.crt \
  --key=path/to/cert.key
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: secure-ingress
spec:
  tls:
  - hosts:
    - food.bar.com
    secretName: food-bar-tls-secret  # Reference to TLS secret
  rules:
  - host: food.bar.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: http-svc
            port:
              number: 80

üß™ Secure Access Demo:

# Fails without certificates
curl -H "Host: food.bar.com" <Ingress_IP>
# Result: Connection requires TLS

# Works with certificates
curl --cacert ca.crt --cert client.crt --key client.key \
     -H "Host: food.bar.com" https://<Ingress_IP>
# Result: Secure access successful

Certificate Storage ‚úÖ Kubernetes Secrets ‚ùå Inline in Route (HAProxy)
Git Storage ‚úÖ Safe for Git ‚ö†Ô∏è Contains sensitive data

‚ùå OpenShift Routes Limitation

HAProxy-based Routes cannot store TLS certificates in Secrets:

# OpenShift Route - certificates inline (not recommended)
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: secure-route
spec:
  host: app.example.com
  tls:
    termination: edge
    certificate: |  # Certificate data directly here - not secure!
      -----BEGIN CERTIFICATE-----
      MIIDXTCCAkWgAwIBAgIJAKZ...
      -----END CERTIFICATE-----
    key: |
      -----BEGIN PRIVATE KEY-----
      MIIEvQIBADANBgkqhkiG9w0...
      -----END PRIVATE KEY-----

‚úÖ Workaround: Use Ingress on OpenShift

Many customers use Kubernetes Ingress even on OpenShift for better secret management:

# Use Ingress on OpenShift for better certificate management
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: openshift-ingress
spec:
  tls:
  - hosts:
    - app.example.com
    secretName: app-tls-secret  # Much better!
  rules:
  - host: app.example.com
    # ... rest of configuration
