🎯 Production vs Development Environments
Critical Truth: Local environments like Minikube are NOT for production!
DevOps Engineers need to know how to manage real production clusters.

🔧 Development Environments

Examples:

  • Minikube - Single node cluster
  • k3s - Lightweight Kubernetes
  • kind - Kubernetes in Docker
  • k3d - k3s in Docker
  • microk8s - Ubuntu's Kubernetes

❌ Limitations:

  • Not full-blown Kubernetes clusters
  • No high availability (HA) support
  • Limited resources (2 CPU, 4GB RAM)
  • Single node operation
  • No production issue resolution

🏭 Production Environments

Requirements:

  • High Availability - Multiple masters
  • Scalability - Multiple worker nodes
  • Reliability - Production-grade components
  • Support - Vendor or community support
  • Security - Enterprise security features

✅ Solutions:

  • Kubernetes distributions
  • Managed services (EKS, AKS, GKE)
  • Enterprise platforms (OpenShift, Rancher)
  • Full Kubernetes installations

⚠️ Interview Reality Check

Never say you used Minikube in production during interviews! Interviewers need to know you understand production cluster management, which is a primary DevOps responsibility.

📦 Understanding Kubernetes Distributions

🐧 Linux Distribution Analogy

Just like Linux has distributions built on the open-source kernel, Kubernetes has distributions built on the open-source container orchestration engine.

Linux Distributions

  • Base: Linux Kernel (Open Source)
  • Examples: Ubuntu, Red Hat, CentOS, Amazon Linux
  • Benefits: Support, security patches, enterprise features
  • Support: Direct vendor support for paid versions

Kubernetes Distributions

  • Base: Kubernetes (Open Source)
  • Examples: OpenShift, Rancher, EKS, Tanzu
  • Benefits: Enterprise features, support, management tools
  • Support: Direct vendor support for issues

🏆 Popular Kubernetes Distributions (Production Order)

1

Kubernetes (Direct Installation)

Open-source Kubernetes installed directly on infrastructure

  • Pros: Cost-effective, full control, flexibility
  • Cons: No instant vendor support
  • Best for: Large organizations with hundreds of clusters
2

Red Hat OpenShift

Enterprise Kubernetes platform with additional tools and support

  • Pros: Enterprise features, Red Hat support, security
  • Cons: Requires Red Hat subscription
  • Best for: Enterprise environments needing support
3

Rancher

Kubernetes management platform for multiple clusters

  • Pros: Multi-cluster management, user-friendly UI
  • Cons: Additional complexity layer
  • Best for: Organizations managing multiple clusters
4

VMware Tanzu

VMware's Kubernetes platform for hybrid cloud

  • Pros: VMware integration, enterprise support
  • Cons: VMware ecosystem dependency
  • Best for: VMware-centric environments
5

Managed Services

EKS (AWS), AKS (Azure), GKE (Google), Docker Kubernetes Engine

  • Pros: Fully managed, cloud provider support
  • Cons: Higher costs, vendor lock-in
  • Best for: Teams wanting minimal operational overhead

💡 Why Organizations Choose Direct Kubernetes

  • Cost: Hundreds of clusters on managed services = significant expenses
  • Flexibility: Custom configurations and specific organizational needs
  • Independence: No vendor lock-in, full control over infrastructure
  • Scale: Thousands of developers across multiple environments
🔧 Introduction to kops (Kubernetes Operations)

🚀 What is kops?

kops stands for Kubernetes Operations - the most popular tool for managing production Kubernetes cluster lifecycle.

🛠️ kubeadm (Traditional)

  • Manual installation process
  • Complex upgrade procedures

  • Limited automation
  • Good for learning basics

⚙️ kops (Production)

  • Complete cluster lifecycle management
  • Automated upgrades and modifications
  • Infrastructure as Code approach

📋 kops Prerequisites & Setup

💸 Cost Warning!

kops creates real AWS resources that incur costs:

  • EC2 instances for master and worker nodes
  • EBS volumes for persistent storage
  • S3 buckets for configuration storage
  • Route 53 entries for custom domains
  • Load balancers and networking components

Tip: You can learn the process without spending by stopping before the final update command!

🛠️ kops Practical Implementation

🪣 Step 1: Create S3 Bucket for kops State

Why S3 Bucket?

kops needs S3 bucket to store cluster configurations and state information for managing hundreds of clusters efficiently.

# Create S3 bucket (bucket name must be globally unique!)
aws s3 mb s3://kops-ab-storage-one

# Note: Replace 'kops-ab-storage-one' with your unique bucket name
# Example alternatives: kops-yourname-storage, kops-company-dev-clusters

⚙️ Step 2: Create Cluster Configuration

💡 Cost-Saving Tip

This command creates configuration only - NO AWS resources yet!
You can safely run this without incurring costs.

# Create cluster configuration
kops create cluster \
    --name=k8s.local \
    --state=s3://kops-ab-storage-one \
    --zones=us-east-1a \
    --node-count=2 \
    --node-size=t2.micro \
    --master-size=t2.micro \
    --ebs-volume-size=8

📊 kops Command Parameters Explained

Parameter Value Description Production Notes
--name k8s.local Cluster name and domain Use company domain in production
--state s3://bucket-name S3 bucket for cluster state One bucket can store multiple clusters
--zones us-east-1a AWS availability zone Use multiple zones for HA
--node-count 2 Number of worker nodes Scale based on workload needs
--node-size t2.micro Worker node instance type Use larger instances in production
--master-size t2.micro Master node instance type Use t3.medium+ for production
--ebs-volume-size 8 EBS volume size in GB Increase based on storage needs

🌐 Step 3: Domain Configuration (Optional)

🏠 Local Domain (.k8s.local)

  • Free to use

  • No Route 53 setup required
  • Interview acceptable for non-prod
# No additional setup needed for .k8s.local

🏢 Custom Domain (company.com)

  • Required for production
  • Professional appearance

# Create hosted zone for custom domain
aws route53 create-hosted-zone \
    --name dev.example.com \
    --caller-reference $(date +%s)

# Replace dev.example.com with your domain

🚨 Step 4: Deploy Cluster (COST ALERT!)

💸 STOP HERE TO AVOID COSTS!

The following command starts creating actual AWS resources and WILL incur charges:

  • EC2 instances (master + 2 workers = ~$30-50/month)
  • EBS volumes (~$5-10/month)
  • Load balancer (~$20/month)
  • Data transfer costs

Only proceed if you have AWS free credits or budget allocated!

# THIS COMMAND STARTS BILLING!
kops update cluster --name k8s.local --yes

# After running, kops will:
# 1. Create EC2 instances
# 2. Set up networking
# 3. Install Kubernetes
# 4. Configure cluster components
# 5. This process takes 10-15 minutes

💰 Expected Monthly Costs (t2.micro setup)

  • Master node (t2.micro): ~$8.50/month
  • Worker nodes (2x t2.micro): ~$17/month
  • EBS volumes (3x 8GB): ~$2.40/month
  • Load balancer: ~$18/month
  • Data transfer: ~$1-5/month
  • Total estimated: ~$47-51/month

Note: Costs vary by region and actual usage patterns.

✅ Post-Deployment Verification

# Verify cluster is running
kubectl get nodes

# Check cluster info
kubectl cluster-info

# Validate all components
kops validate cluster --name k8s.local

# Get cluster details
kubectl get pods --all-namespaces

🗑️ Cleanup (Important for Cost Control)

💡 Always Clean Up!

Don't forget to delete your cluster when done experimenting to avoid ongoing charges.

# Delete the entire cluster and all AWS resources
kops delete cluster --name k8s.local --yes

# This will remove:
# - All EC2 instances
# - EBS volumes  
# - Load balancers
# - Security groups
# - All associated resources

🏢 Production Considerations

🏗️ High Availability Setup

  • Multiple master nodes (odd number: 3, 5)
  • Multiple availability zones
  • Etcd backup strategy
  • Load balancer redundancy

🔒 Security Hardening

  • RBAC configuration
  • Network policies
  • Pod security policies
  • Regular security updates

📊 Monitoring & Logging

  • Prometheus for metrics
  • Grafana for dashboards
  • ELK stack for logging
  • Alertmanager for notifications

💾 Backup & Disaster Recovery

  • Etcd backup automation
  • Application data backups
  • Cross-region replication
  • Recovery testing procedures

❓ Cluster Lifecycle

Question: "How do you manage cluster lifecycle?"

❓ High Availability

Question: "How do you ensure cluster high availability?"

Answer: "Multiple master nodes across different AZs, etcd clustering, load balancer redundancy, and automated backup strategies."

❓ Cost Optimization

Question: "How do you optimize Kubernetes costs?"

Answer: "Using spot instances for dev environments, right-sizing nodes, implementing cluster autoscaling, and using direct Kubernetes instead of managed services for large deployments."
