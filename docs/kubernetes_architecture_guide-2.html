⚖️ Docker vs Kubernetes: Component Comparison
Key Insight: Docker's simplest unit = Container
Kubernetes' simplest unit = Pod

🐳 Docker Container Creation

1
User Action: Write Dockerfile & build image
2
Run Command: docker run image_name
3
Requirement: Container Runtime (Docker shim)
4
Result: Container runs on single host 
# Docker container creation
docker build -t my-app .
docker run -d --name my-container my-app

# Single host limitation
docker ps  # Shows containers on this host only

☸️ Kubernetes Pod Creation

1
User Action: Request goes to Master (Control Plane)
2
API Server: Receives request & processes
3
Scheduler: Decides which Worker Node
4
Worker Node: kubelet + Container Runtime 
# Kubernetes Pod creation
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  containers:
  - name: my-container
    image: my-app:latest

# Apply to cluster
kubectl apply -f pod.yaml
🔧 Worker Node (Data Plane) Components

💻 Data Plane - Where Your Applications Run

The execution environment for your containers and Pods

🤖 kubelet

Primary Role: Pod lifecycle management

  • Ensures Pods are always running
  • Monitors Pod health continuously
  • Reports Pod status to Control Plane
  • Triggers auto-healing when needed
# kubelet responsibilities
- Pod creation & deletion
- Health monitoring
- Resource allocation
- Status reporting

🌐 kube-proxy

Primary Role: Networking & Load Balancing

  • Assigns IP addresses to Pods
  • Provides default load balancing
  • Manages network routing
  • Uses Linux iptables for configuration
# kube-proxy functions
- IP address management
- Load balancing (50%-50% traffic split)
- Network proxy services
- iptables rule management

📦 Container Runtime

Primary Role: Container execution

  • Actually runs containers inside Pods
  • Supports multiple runtimes
  • Implements Container Runtime Interface (CRI)
  • Docker is NOT mandatory!

🔄 Supported Container Runtimes

  • containerd - Most popular
  • CRI-O - Red Hat optimized
  • Docker shim - Traditional Docker
  • Any CRI-compliant runtime

✅ Worker Node Summary

With these three components, you have everything needed to run applications on a Worker Node. The Worker Node is the execution environment where your actual workloads run.

🎛️ Master Node (Control Plane) Components

🧠 Control Plane - The Brain of Kubernetes

Makes all the decisions and manages the entire cluster

❓ Why Do We Need a Control Plane?

For enterprise-level tools, you need centralized management for:

  • 🎯 Decision Making: Where should Pods be scheduled?
  • 🔒 Security & Identity: Who can access what?
  • 📋 Request Handling: Processing user requests
  • 🏗️ Cluster Management: Maintaining desired state

🚪 API Server

Role: The Heart of Kubernetes

  • Central entry point for all requests
  • Exposes Kubernetes cluster to external world
  • Handles authentication & authorization
  • Validates and processes API requests
# All kubectl commands go through API Server
kubectl get pods
kubectl create deployment nginx --image=nginx
kubectl delete pod my-pod

# API Server validates and processes these

📅 Scheduler

Role: Pod Placement Decision Maker

  • Decides which Worker Node gets which Pod
  • Considers resource requirements
  • Evaluates node capacity and constraints
  • Optimizes cluster resource utilization
# Scheduler considers:
- Node resource availability (CPU, Memory)
- Pod resource requirements
- Node affinity rules
- Taints and tolerations
- Anti-affinity rules

🗄️ etcd

Role: Cluster State Database

  • Distributed key-value store
  • Stores ALL cluster information
  • Critical for disaster recovery
  • Must be backed up regularly

💾 etcd Importance

Without etcd, you lose your entire cluster state! It stores:

  • Pod definitions and status
  • Service configurations
  • Secrets and ConfigMaps
  • Cluster metadata

🎮 Controller Manager

Role: Maintains Desired State

  • Manages Kubernetes built-in controllers
  • Ensures Replica Sets maintain desired count
  • Handles auto-scaling operations
  • Monitors and corrects cluster state

🔄 Key Controllers

  • Replication Controller: Maintains Pod replicas
  • Deployment Controller: Manages deployments
  • Service Controller: Manages services
  • Node Controller: Monitors node health

☁️ Cloud Controller Manager (CCM)

Role: Cloud Provider Integration

  • Translates Kubernetes requests to cloud APIs
  • Manages cloud-specific resources
  • Handles load balancer creation
  • Manages cloud storage integration
# Cloud integrations handled by CCM:
- AWS EKS (Elastic Kubernetes Service)
- Azure AKS (Azure Kubernetes Service)  
- GCP GKE (Google Kubernetes Engine)
- Custom cloud providers

🏠 On-Premise Note

If running Kubernetes on-premise, Cloud Controller Manager is NOT required!

🔄 Pod Creation Workflow

📋 Step-by-Step Pod Creation Process

User Request
API Server
etcd Store
Scheduler
kubelet
Container Runtime
Pod Running
1
User submits Pod YAML: kubectl apply -f pod.yaml
2
API Server receives request: Validates authentication, authorization, and Pod specification
3
etcd stores Pod definition: Pod spec saved in cluster database
4
Scheduler decides placement: Evaluates Worker Nodes and selects optimal node
5
kubelet creates Pod: kubelet on selected Worker Node receives instruction
6
Container Runtime starts containers: Pulls images and starts containers inside Pod
7
kube-proxy configures networking: Assigns IP and configures routing 
# Example Pod YAML
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
spec:
  containers:
  - name: nginx-container
    image: nginx:latest
    ports:
    - containerPort: 80
    resources:
      requests:
        memory: "128Mi"
        cpu: "100m"
      limits:
        memory: "256Mi"
        cpu: "200m"

🎯 Key Insight

Notice how the Control Plane makes decisions (API Server, Scheduler) while the Data Plane executes (kubelet, Container Runtime). This separation provides the enterprise-level orchestration capabilities!
