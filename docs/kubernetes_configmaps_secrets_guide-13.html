🎯 Problem Statement: Application Configuration

🏗️ Real-World Scenario

Consider a backend application that connects to a database to retrieve and provide information to users.

Application Configuration Requirements

Backend App
Database Connection
User Data

Required Configuration:

  • Database Port (e.g., 3306)
  • Database Username
  • Database Password
  • Connection Type/Pool Size

❌ The Hardcoding Problem

Hardcoding configuration in application code is problematic:

  • Configuration changes require code changes
  • Different environments need different values
  • Sensitive data exposed in source code
  • Application fails when configuration changes

💡 Traditional Solutions (Outside Kubernetes)

🌐 Environment Variables

# Traditional approach
export DB_PORT=3306
export DB_USERNAME=admin
export DB_PASSWORD=secret123

# Application reads using OS modules
import os
db_port = os.getenv('DB_PORT')

📁 Configuration Files

# Config file approach
/etc/app/config.properties:
db.port=3306
db.username=admin
db.password=secret123

# Application reads from file
Properties props = new Properties();
props.load(new FileInputStream("/etc/app/config.properties"));

☸️ Kubernetes Challenge

In Kubernetes containers:

  • Users typically don't have login access to containers
  • Configuration values might change frequently
  • Values might not be known when building Docker images
  • Need centralized configuration management

Solution: Kubernetes provides ConfigMaps and Secrets!

📋 ConfigMaps: Non-Sensitive Configuration

🎯 What are ConfigMaps?

ConfigMaps are Kubernetes resources used to store non-sensitive configuration data as key-value pairs.

📊 ConfigMap Characteristics

📋 ConfigMap Details

Suitable for:
  • Database port numbers
  • Application feature flags
  • API endpoints
  • Timeout configurations
  • Environment-specific settings
Storage:
  • Stored in etcd
  • NOT encrypted at rest
  • Easily accessible via kubectl

⚠️ NOT Suitable for:

Sensitive data:
  • Passwords
  • API keys
  • Database credentials
  • TLS certificates
  • OAuth tokens
Security Risk:
  • Unencrypted in etcd
  • Visible in kubectl describe
  • Easily accessible to anyone with cluster access

🔄 How ConfigMaps Work

Create ConfigMap
Store in etcd
Mount in Pod
App Uses Config

Two Ways to Use ConfigMaps in Pods:

🌐 Environment Variables

Inject as environment variables inside container

Limitation: Changes require pod restart

📁 Volume Mounts

Mount as files in container filesystem

Advantage: Updates reflect automatically

🔓 Security Limitation

ConfigMap data is NOT encrypted at rest in etcd. Anyone with access to etcd or kubectl can easily retrieve the information:

# Easy access to ConfigMap data
kubectl describe configmap my-config
kubectl get configmap my-config -o yaml

# Data is visible in plain text!
🔐 Secrets: Sensitive Configuration Data
Core Purpose: Secrets solve the same problem as ConfigMaps but specifically handle sensitive data with enhanced security features.

🛡️ Why Secrets Are Needed

⚠️ The Security Risk

Storing sensitive data like passwords in ConfigMaps is a major security risk because:

  • Data saved unencrypted in etcd
  • Hackers with etcd access can compromise entire application
  • Plain text exposure in kubectl commands
  • No access control for sensitive information

🔒 Secret Security Features

🔐 Encryption at Rest

Data encrypted before storage in etcd

Custom encryption mechanisms supported

Useless without decryption key

👥 Strong RBAC

Role-Based Access Control

Least privilege principle

Restrict Secret access to minimal users

👁️ Access Control

Grant ConfigMap, Pod, Deployment access

Explicitly deny Secret access

Granular permission management

🛡️ Multi-Layer Protection

Secrets provide protection against two attack vectors:

  1. etcd Access: Data encrypted, useless without key
  2. kubectl Access: Strong RBAC prevents resource access

⚖️ ConfigMap vs Secret Comparison

Aspect ConfigMap Secret
Purpose Non-sensitive configuration Sensitive configuration
Data Examples DB port, API endpoints, feature flags Passwords, API keys, certificates
Encryption at Rest ❌ Not encrypted ✅ Encrypted in etcd
RBAC Requirement Standard access control Strong RBAC recommended
Visibility Plain text in kubectl Base64 encoded display
Use Cases Environment config, app settings Database credentials, TLS certs

📝 Step 1: Create ConfigMap YAML

1

Create ConfigMap YAML File

# Create cm.yaml file
vim cm.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: test-cm
data:
  DB-Port: "3306"  # Data stored as key-value pairs
YAML Structure Explanation:
  • apiVersion: v1 - Standard for ConfigMaps
  • kind: ConfigMap - Resource type
  • metadata.name - ConfigMap name for reference
  • data - Key-value pairs section
2

Create ConfigMap in Cluster

# Apply ConfigMap (always use apply over create)
kubectl apply -f cm.yaml
configmap/test-cm created
# Verify creation
kubectl get cm

# Describe ConfigMap to see data
kubectl describe cm test-cm

Namespace:    default
Labels:       <none>
Annotations:  <none>

🌐 Demo: ConfigMap as Environment Variables

🎯 Goal

Take the DB-Port value from ConfigMap and make it an environment variable inside a pod.

1

Modify Deployment YAML

Using existing deployment.yaml for Python Django application:

# Add env section inside containers specification
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sample-python-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: sample-python-app
  template:
    metadata:
      labels:
        app: sample-python-app
    spec:
      containers:
      - name: python-app
        image: abhishekf5/simple-python-app:latest
        ports:
        - containerPort: 8000
        env:                              # Add environment variables
        - name: DB_Port                   # Environment variable name
          valueFrom:
            configMapKeyRef:              # Reference ConfigMap (not configMapRef!)
              name: test-cm               # ConfigMap name
              key: DB-Port                # Key from ConfigMap data
⚠️ Common Mistake Alert!

Initial attempt might use configMapRef - this is incorrect!

Correct syntax: configMapKeyRef

Always follow official documentation!

2

Apply Modified Deployment

# Apply updated deployment
kubectl apply -f deployment.yaml

This terminates existing pods and creates new ones with updated configuration.

# Watch pods being recreated
kubectl get pods -w

# Wait for Running status
kubectl get pods
NAME                               READY   STATUS    RESTARTS   AGE
sample-python-app-xxx-xxx          1/1     Running   0          45s
sample-python-app-yyy-yyy          1/1     Running   0          45s
3

Verify Environment Variable

# Get pod name
kubectl get pods

# Exec into pod (need -it for interactive terminal)
kubectl exec -it sample-python-app-xxx-xxx -- /bin/bash

# Check environment variables inside pod
env | grep DB
DB_Port=3306
🎉 Success!

ConfigMap value successfully injected as environment variable!

🔄 Environment Variable Limitation

⚠️ Problem with Updating Environment Variables

Let's test what happens when ConfigMap is updated:

1
Update ConfigMap
# Edit cm.yaml, change 3306 to 3307
vim cm.yaml

# Apply the change
kubectl apply -f cm.yaml
2
Check Same Pod
# Exec into SAME pod again
kubectl exec -it sample-python-app-xxx-xxx -- /bin/bash

# Check environment variable
env | grep DB
DB_Port=3306  # Still old value!

Issue: Environment variables inside containers are immutable. Pod needs to be recreated to get updated values.

Problem in Production: Restarting containers can lead to traffic loss and downtime.

📁 Demo: ConfigMap as Volume Mount (Files)

💡 Volume Mount Advantage

Mount ConfigMap data as files inside the pod. This allows pods to pick up ConfigMap changes without restarting!

1

Modify Deployment for Volume Mount

apiVersion: apps/v1
kind: Deployment
metadata:
  name: sample-python-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: sample-python-app
  template:
    metadata:
      labels:
        app: sample-python-app
    spec:
      containers:
      - name: python-app
        image: abhishekf5/simple-python-app:latest
        ports:
        - containerPort: 8000
        # Remove env section, add volumeMounts
        volumeMounts:
        - name: db-config-volume          # Volume mount name
          mountPath: /opt/app/config      # Path inside container
      volumes:                            # Add at pod spec level
      - name: db-config-volume            # Volume name
        configMap:                        # Volume source
          name: test-cm                   # ConfigMap name
YAML Structure Explanation:
  • volumes: Defined at pod template spec level
  • volumeMounts: Defined inside container spec
  • Volume references ConfigMap: Gets data from test-cm
  • Mount path: Where files appear inside container
2

Apply Modified Deployment

# Apply volume mount configuration
kubectl apply -f deployment.yaml

# Watch pods being recreated
kubectl get pods -w
NAME                               READY   STATUS    RESTARTS   AGE
sample-python-app-zzz-zzz          1/1     Running   0          60s
3

Verify Volume Mount

# Exec into new pod
kubectl exec -it sample-python-app-zzz-zzz -- /bin/bash

# Verify environment variable is gone
env | grep DB
# (should show nothing)

# Navigate to mounted path
cd /opt/app/config

# List files (file named after ConfigMap key)
ls
DB-Port
# Read file content
cat DB-Port
3306
🎉 Success!

ConfigMap data mounted as file! File name = ConfigMap key, File content = ConfigMap value

Test automatic file updates without pod restart:

1
Update ConfigMap
# Edit ConfigMap directly

# Verify update

🔄 Continuous Synchronization

Kubernetes continuously reads ConfigMap and updates mounted files automatically. This enables:

  • Zero-downtime configuration updates
  • Dynamic application reconfiguration
  • Better production environment management
🔐 Demo: Creating and Using Secrets

🎯 Secret Behavior

Secrets behave similarly to ConfigMaps for:

  • Mounting as files
  • Exposing as environment variables
  • Dynamic updates (for volume mounts)

Key Difference: Enhanced security with encryption and access control

🗂️ Secret Types

🔑 Generic

Key-value pairs

Database credentials

🌐 TLS

TLS certificates

HTTPS configurations

🐳 Docker Registry

Image pull secrets

Private registry access

🚀 Creating Generic Secret

1

Method 1: kubectl create command

# Create generic secret with literal value
kubectl create secret generic test-secret --from-literal=DB-Port=3306

# Note: Storing DB-Port in Secret is just for demo
# In reality, you'd store passwords/usernames
secret/test-secret created
2

Verify Secret Creation

# List secrets
kubectl get secrets
NAME                  TYPE                                  DATA   AGE
default-token-xxxxx   kubernetes.io/service-account-token   3      10d
test-secret          Opaque                                1      30s
# Describe secret (notice the difference from ConfigMap)
kubectl describe secret test-secret
Name:         test-secret
Namespace:    default
Labels:       <none>
Annotations:  <none>

Type:  Opaque

Data
====
DB-Port:  4 bytes  # Shows size, NOT the actual value!
🔒 Security Feature

Notice: kubectl describe shows data size but NOT the actual value!

3

View Secret Data (Base64 Encoded)

# Edit secret to see encoded data
kubectl edit secret test-secret
Secret YAML Content:
apiVersion: v1
data:
  DB-Port: MzMwNg==  # Base64 encoded value
kind: Secret
metadata:
  name: test-secret
  namespace: default
type: Opaque
Decode Base64 (Outside Kubernetes):
# Decode the Base64 value
echo "MzMwNg==" | base64 --decode
3306
⚠️ Important Note

Base64 is just encoding, not encryption! The actual encryption happens when Kubernetes stores the Secret in etcd.

📝 Method 2: YAML File Creation

# secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: test-secret-yaml
type: Opaque
data:
  DB-Username: YWRtaW4=      # admin (base64 encoded)
  DB-Password: c2VjcmV0MTIz  # secret123 (base64 encoded)

# To encode values:
# echo -n "admin" | base64
# echo -n "secret123" | base64

🔧 Using Secrets in Deployments

🌐 Secret as Environment Variable

apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-with-secret
spec:
  template:
    spec:
      containers:
      - name: my-app
        image: my-app:latest
        env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:        # Note: secretKeyRef not configMapKeyRef
              name: test-secret
              key: DB-Password

📁 Secret as Volume Mount

apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-with-secret
spec:
  template:
    spec:
      containers:
      - name: my-app
        image: my-app:latest
        volumeMounts:
        - name: secret-volume
          mountPath: /etc/secrets
      volumes:
      - name: secret-volume
        secret:                  # Note: secret not configMap
          secretName: test-secret

🔄 Secret Usage Pattern

Secrets follow the same usage patterns as ConfigMaps:

  • Environment variables: secretKeyRef
  • Volume mounts: secret volume type
  • Dynamic updates work for volume mounts
  • Environment variables still require pod restart

🔍 Troubleshooting & Best Practices

🚨 Common Issues & Solutions

❌ Validation Error

Error: "unknown field configMapRef"

Solution: Use configMapKeyRef not configMapRef

Always follow documentation!

❌ Environment Not Updated

Issue: ConfigMap changed but env var same

Reason: Environment variables are immutable

Solution: Use volume mounts for dynamic updates

❌ Mount Path Issues

Issue: Files not visible in container

Check: Volume mount path exists

Check: Volume name matches

❌ Base64 Confusion

Issue: Secret values garbled

Remember: Kubernetes auto-decodes

Don't decode in application code

✅ Best Practices

🔐 Secret Management

  • Use Secrets for sensitive data only
  • Implement strong RBAC
  • Consider external secret managers
  • Regular secret rotation

📋 ConfigMap Usage

  • Use for non-sensitive configuration
  • Prefer volume mounts for dynamic updates
  • Organize by environment/component
  • Version control your YAML files

🔄 Update Strategies

  • Volume mounts: Automatic updates
  • Environment variables: Manual restart
  • Test configuration changes
  • Monitor application behavior
