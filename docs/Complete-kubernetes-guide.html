🐳 Docker vs Kubernetes: The Complete Comparison

Docker is a container platform, while Kubernetes is a container orchestration platform that solves Docker's limitations at scale.

🚨 Docker Problems in Production

  • Single Host Limitation: All containers run on one machine - if it fails, everything goes down
  • No Auto-Healing: When containers crash, they stay crashed until manually restarted
  • No Auto-Scaling: Cannot automatically handle increased traffic loads
  • Limited Enterprise Features: Missing built-in load balancers, firewalls, API gateways
  • Resource Conflicts: Containers compete for resources on the same host

✅ Kubernetes Solutions

  • Cluster Architecture: Multiple nodes provide high availability and fault tolerance
  • Auto-Healing: Automatically restarts failed containers and replaces unhealthy nodes
  • Auto-Scaling: HPA and VPA automatically scale based on CPU, memory, or custom metrics
  • Enterprise-Grade: Built-in load balancing, security policies, and networking
  • Resource Management: Intelligent scheduling and resource allocation across nodes

🏗️ Kubernetes Architecture Deep Dive

Kubernetes follows a master-worker architecture with clear separation of concerns between control and data planes.

🧠 Control Plane (Master Node)

Makes decisions about the cluster

🌐 API Server

Role: Entry point for all cluster operations

Function: Validates requests, manages authentication, exposes REST API

Communication: All components talk through API server

📅 Scheduler

Role: Decides which node should run each pod

Function: Considers resource requirements, node capacity, and constraints

Algorithm: Uses scoring to find the best node

🗄️ etcd

Role: Distributed key-value database

Function: Stores all cluster state and configuration

Importance: Single source of truth for cluster data

🎛️ Controller Manager

Role: Runs various controllers

Function: Maintains desired state (ReplicaSet, Deployment, Service controllers)

Process: Continuously watches and reconciles actual vs desired state

☁️ Cloud Controller Manager

Role: Integrates with cloud provider APIs

Function: Manages cloud-specific resources (LoadBalancers, Volumes)

Examples: AWS ELB, GCP Load Balancer, Azure Load Balancer

⚙️ Worker Node (Data Plane)

Runs the actual workloads

🤖 Kubelet

Role: Node agent that manages pods

Function: Receives pod specs from API server and ensures containers are running

Health: Performs health checks and reports status back to API server

🌐 Kube-proxy

Role: Network proxy running on each node

Function: Maintains network rules and enables Service communication

Implementation: Uses iptables or IPVS for traffic routing

🐳 Container Runtime

Role: Actually runs containers

Options: Docker, containerd, CRI-O

Interface: Implements Container Runtime Interface (CRI)

📦 Pods, Deployments & ReplicaSets

Understanding the Pod Concept

A Pod is the smallest deployable unit in Kubernetes. Think of it as a "wrapper" around your containers that provides:

  • Shared IP address for all containers in the pod
  • Shared storage volumes
  • Shared network namespace (containers can communicate via localhost)
  • Lifecycle management (all containers start/stop together)
Basic Pod YAML Configuration
# pod.yaml - Basic Pod Definition apiVersion: v1 kind: Pod metadata: name: nginx-pod labels: app: nginx environment: production spec: containers: - name: nginx-container image: nginx:1.21 ports: - containerPort: 80 protocol: TCP resources: requests: memory: "64Mi" cpu: "50m" limits: memory: "128Mi" cpu: "100m" env: - name: ENVIRONMENT value: "production"

⚠️ Why Never Use Pods Directly in Production

Problems with standalone Pods:

  • No auto-healing - if pod dies, it stays dead
  • No auto-scaling - cannot handle traffic increases
  • No rolling updates - deployments cause downtime
  • No replica management - single point of failure

🧪 Auto-Healing Demonstration

Try this experiment:

  1. Create deployment with 3 replicas: kubectl apply -f deployment.yaml
  2. Open two terminals - watch pods in one: kubectl get pods -w
  3. Delete a pod in the other: kubectl delete pod <pod-name>
  4. Watch the ReplicaSet immediately create a new pod to maintain desired state!

This demonstrates Kubernetes' powerful self-healing capability in action.

🔄 How Auto-Healing Works

1

Pod fails or gets deleted

Container crashes, node fails, or manual deletion occurs

2

Kubelet detects failure

Node agent notices pod is no longer running

3

API Server gets notification

Kubelet reports pod status change to control plane

4

ReplicaSet Controller takes action

Detects actual replicas (2) < desired replicas (3)

5

New pod gets scheduled

Scheduler finds available node and creates replacement pod

6

Desired state restored

Application continues running with no service interruption

🌐 Services: Load Balancing & Service Discovery

The Problem Services Solve

Imagine you have 3 replica pods running your application. Each pod has a different IP address (10.1.1.5, 10.1.1.8, 10.1.1.12). When a pod restarts due to auto-healing, it gets a new IP address. How do users and other services reliably connect to your application?

❌ Without Services

  • Pods have dynamic, changing IP addresses
  • No load balancing between replicas
  • Manual service discovery required
  • Applications break when pods restart
  • No stable endpoint for external access

✅ With Services

  • Stable virtual IP that never changes
  • Automatic load balancing across pods
  • DNS-based service discovery
  • Seamless handling of pod changes
  • Multiple exposure options (internal/external)

Access Level: External via Node IP + Port

Use Cases:

  • Development and testing
  • Internal company applications
  • Admin dashboards
  • Monitoring interfaces

Port Range: 30000-32767 (automatically assigned if not specified)

🌍 LoadBalancer

Access Level: Public internet

Use Cases:

  • Production web applications
  • Public APIs
  • E-commerce sites
  • Customer-facing services

Access: http://<external-lb-ip>

Cloud Integration: Creates AWS ELB, GCP Load Balancer, or Azure Load Balancer

🔐 Kubernetes Security Best Practices

Security in Kubernetes requires a layered approach covering cluster, network, workload, and data protection.

🔑 API Server Security

The API Server is the heart of Kubernetes - securing it is critical.

  • TLS Encryption: All communications encrypted
  • Authentication: x509 certificates, OIDC, or bearer tokens
  • Authorization: RBAC policies
  • Admission Control: Validate and mutate requests
  • Audit Logging: Track all API access

👥 RBAC (Role-Based Access Control)

Principle of least privilege - give users minimum required permissions.

  • Roles: Define what actions are allowed
  • RoleBindings: Assign roles to users/groups
  • ClusterRoles: Cluster-wide permissions
  • Service Accounts: Identity for pods

🛡️ Network Policies

Micro-segmentation controls pod-to-pod communication.

  • Ingress Rules: Control incoming traffic
  • Egress Rules: Control outgoing traffic
  • Default Deny: Block all traffic by default
  • CNI Support: Requires Calico, Cilium, or Weave

🔒 Data Protection

Protect sensitive data at rest and in transit.

  • Secrets Management: Encrypted storage for passwords/keys
  • Encryption at Rest: etcd data encryption
  • Image Security: Scan for vulnerabilities
  • Pod Security: Security contexts and policies

⚠️ Common Security Mistakes

  • Running containers as root: Always use non-root users
  • Overprivileged service accounts: Apply principle of least privilege
  • No network policies: Default allows all pod-to-pod communication
  • Unscanned images: Use images from trusted registries and scan regularly
  • Secrets in plain text: Never put secrets in container images or YAML files
  • No resource limits: Pods can consume all node resources

KOPS: Kubernetes Operations Tool

KOPS is the most popular tool for managing production Kubernetes clusters on AWS. It handles the complete lifecycle:

  • Installation: Creates cluster infrastructure
  • Upgrades: Kubernetes version upgrades
  • Scaling: Add/remove nodes
  • Configuration: Cluster parameter changes
  • Deletion: Clean cluster removal

🎯 Top Kubernetes Interview Questions & Answers

What is the difference between Docker and Kubernetes?
Docker is a container platform that packages applications into containers.
Kubernetes is a container orchestration platform that manages Docker containers at scale.

Key differences:
• Docker runs on single host, Kubernetes manages clusters
• Docker lacks auto-healing/scaling, Kubernetes provides both
• Docker is for development, Kubernetes is for production
• Kubernetes solves Docker's limitations with enterprise features
Explain Kubernetes architecture and its main components
Control Plane (Master Node):
• API Server: Entry point for all cluster operations
• Scheduler: Assigns pods to appropriate nodes
• etcd: Key-value store for cluster state
• Controller Manager: Manages controllers (ReplicaSet, Deployment)
• Cloud Controller Manager: Integrates with cloud providers

Worker Nodes (Data Plane):
• Kubelet: Manages pods on the node
• Kube-proxy: Handles networking and load balancing
• Container Runtime: Runs containers (Docker, containerd, CRI-O)
What is the difference between Pod and Deployment?
Pod:
• Smallest deployable unit in Kubernetes
• Wrapper around one or more containers
• No auto-healing or auto-scaling capabilities
• Ephemeral - when deleted, it's gone forever

Deployment:
• Higher-level abstraction that manages ReplicaSets
• Provides auto-healing and auto-scaling
• Enables rolling updates and rollbacks
• Ensures desired number of pod replicas are always running

Production rule: Never create pods directly, always use Deployments!
Explain the three types of Kubernetes Services
ClusterIP (Default):
• Internal cluster access only
• Use for: Database connections, internal APIs
• Access: service-name.namespace.svc.cluster.local

NodePort:
• External access via Node IP:Port
• Use for: Development, testing, internal apps
• Access: http://node-ip:30000-32767

LoadBalancer:
• Public internet access
• Use for: Production web applications
• Creates cloud provider's load balancer (ELB, GCP LB, Azure LB)
What is the role of Kubelet in Kubernetes?
Kubelet is the node agent that runs on every worker node:

Primary responsibilities:
• Manages pod lifecycle on the node
• Communicates with API server
• Pulls container images and starts containers
• Performs health checks (liveness/readiness probes)
• Reports pod and node status back to control plane
• Mounts volumes and handles secrets/configmaps

Auto-healing process:
1. Kubelet detects pod failure
2. Reports to API server
3. ReplicaSet controller creates new pod
4. Kubelet starts the replacement pod
How does auto-healing work in Kubernetes?
Auto-healing ensures desired state is maintained:

Process flow:
1. Detection: Kubelet notices pod failure
2. Reporting: Status sent to API server
3. Controller action: ReplicaSet detects actual < desired replicas
4. Scheduling: New pod scheduled on available node
5. Recovery: Application continues without downtime

Types of failures handled:
• Container crashes
• Node failures
• Health check failures
• Resource exhaustion

Controllers involved: ReplicaSet, Deployment, StatefulSet, DaemonSet
What is a Namespace in Kubernetes and why is it important?
Namespace provides logical isolation of resources within a cluster:

Benefits:
• Multi-tenancy: Different teams can use same cluster
• Resource isolation: Prevent naming conflicts
• RBAC boundaries: Control access per namespace
• Resource quotas: Limit CPU/memory per namespace
• Network policies: Control traffic between namespaces

Common namespaces:
• default: Default namespace for resources
• kube-system: Kubernetes system components
• kube-public: Publicly accessible resources
• Custom: production, staging, development

What is the difference between Docker Swarm and Kubernetes?
Docker Swarm:
• Docker-native orchestration
• Simple to set up and use
• Limited scaling and networking options
• Best for: Small scale, simple applications
• Less community support and ecosystem

Kubernetes:
• Industry-standard orchestration platform
• Complex but powerful feature set
• Advanced networking, storage, and security
• Best for: Enterprise, large-scale production
• Huge community and ecosystem (CNCF)
• Extensible with Custom Resource Definitions

Market reality: 10/10 DevOps jobs require Kubernetes knowledge
How do you troubleshoot a failing pod in Kubernetes?
Systematic troubleshooting approach:

1. Check pod status:
kubectl get pods - Check current status
kubectl describe pod <pod-name> - Detailed events and conditions

2. Examine logs:
kubectl logs <pod-name> - Current container logs
kubectl logs <pod-name> --previous - Previous container logs

3. Check resources:
kubectl top pod <pod-name> - Resource usage
Check resource limits and requests in pod spec

4. Verify configuration:
Check image availability, secrets, configmaps
Verify environment variables and volume mounts

5. Network troubleshooting:
Test connectivity with kubectl exec -it <pod> -- /bin/bash
Check service endpoints and DNS resolution
Explain the concept of ReplicaSet and how it differs from Deployment
ReplicaSet:
• Ensures specified number of pod replicas are running
• Basic controller that maintains desired state
• Handles pod creation and deletion
• No built-in update strategy

Deployment:
• Higher-level abstraction that manages ReplicaSets
• Provides rolling updates and rollback capabilities
• Creates new ReplicaSet for each update
• Manages multiple ReplicaSet versions

Hierarchy: Deployment → ReplicaSet → Pod

Practical difference:
• ReplicaSet: Ensures replicas but no update strategy
• Deployment: Handles updates without downtime
• Always use Deployment in production for update capabilities

  • Using Pods directly: Always use Deployments for production
  • No resource limits: Set CPU and memory limits on containers

  • No backup strategy: Plan for etcd backups and disaster recovery
